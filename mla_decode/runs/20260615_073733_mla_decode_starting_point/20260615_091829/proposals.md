## Iteration 1

## STATE
This is the start of epoch 2, with only a single baseline at 2612 µs — no other approaches have been tried yet. The roofline SOL targets are ~211 µs (4096 prefill) and ~281 µs (6087 prefill), putting the baseline at roughly 12× above target. The dominant cost at large prefill is the attention phase: `Q @ K^T` and softmax `@ V` over 4096–6144 tokens, with bs=128 and 128 heads. The existing code does scattered per-row softmax in Triton but the core matmuls still run as individual PyTorch ops. There is no overlap, no Flash-Attention style fused attention kernel.

## RATIONALE
The single largest opportunity is replacing the multi-step attention sequence (scores_nope matmul → scores_rope matmul → fused-add-softmax → weighted-sum matmul) with a fused Flash-Attention kernel. This collapses four HBM round-trips (two score writes, one softmax write, one weighted-sum input read) into one streaming pass over the KV cache. At seq_len=4096–6144 the KV cache tensor is ~128 × 6144 × 512 × 2 bytes ≈ 800 MB+; reading it four separate times is the dominant latency. The problem has a non-standard structure: `k_nope` is implicitly `kv_nope @ wK^T` and `v` is implicitly `kv_nope @ wV^T`, but since `kv_nope` is what's actually stored, we can apply the absorption trick differently. The most practical route is to use `flash_attn_func` (from the `flash_attn` package, which is available on H200 installations) or `torch.nn.functional.scaled_dot_product_attention` (SDPA/cuDNN FlashAttention) with appropriate tensor reshaping. SDPA with the `enable_flash` backend would handle the fused attention natively without writing a Triton kernel from scratch, and would deliver near-optimal memory access. The `k_rope` path is already pre-expanded; the challenge is the absorbed `q_nope_latent` and `M` paths. The best approach is to construct explicit Q, K, V tensors of proper shape for SDPA — even if that means materializing `k_nope = kv_nope @ wK^T` explicitly — and let SDPA fuse the softmax + weighted sum in one kernel.

## PROPOSAL
Replace the four-op attention sequence with **`torch.nn.functional.scaled_dot_product_attention` (SDPA)**. Concretely: materialize full K (concatenating k_nope and k_rope along the head-dim axis) and full V by absorbing `wUKV` into explicit per-token projections, then call `F.scaled_dot_product_attention(Q, K, V)` which will dispatch to cuDNN FlashAttention on H200. This collapses the entire attention phase into a single fused kernel that streams the KV cache once rather than four times, dramatically cutting HBM traffic. Ensure tensors are laid out in the `[bs, n_heads, seq_len, head_dim]` format SDPA expects, and contiguous. Also verify that the LoRA projections (`wDQ`, `wDKV`, `wUQ`, `wUKV`, `wO`) are dispatched as contiguous FP8 or BF16 cuBLAS GEMMs — no fragmentation. If `flash_attn` is available as a package, try `flash_attn_with_kvcache` as an alternative path that natively handles the decode pattern (single query token against a long KV cache).

## Iteration 2

## STATE
We have one baseline at 2612 µs and one catastrophic failure (227,341 µs) from expanding full per-head K/V tensors. The worker's diagnosis is exactly right: materializing `[bs=128, nh=128, kv_len=6144, d_nope+d_rope=192]` per-head K allocates ~128×128×6144×192×2 ≈ 38 GB of intermediate tensor, which destroyed performance. The absorption trick in the baseline — keeping queries in latent space (`q_nope_latent [bs, nh, dkv=512]`) and scoring against raw `kv_nope [bs, kv_len, dkv]` — is the correct memory-efficient formulation and must be preserved. The gap from baseline to SOL is still roughly 10–12×.

## RATIONALE
The critical insight is that the attention bottleneck here is **not** a standard QKV attention that SDPA can directly accelerate, because K is implicitly `kv_nope @ wK^T` (not stored). The absorbed formulation replaces K-expansion with a single smaller Q-projection: `q_nope_latent [bs, nh, dkv]` scores against `kv_nope [bs, kv_len, dkv]` — this is a batched GEMM of shape `[bs×nh, 1, dkv] × [bs, dkv, kv_len]`, i.e., a **GEMVs** (each batch element, each head computes one dot product per token). The dominant cost becomes `bs × nh` independent dot products over `kv_len` elements of dimension `dkv=512`. With bs=128 and nh=128, this is 16,384 rows × kv_len=6144 cols — a pure memory-bandwidth bound operation. The current code uses `torch.matmul` which goes through cuBLAS; this is not ideal for this GEMVs shape. The key opportunity is to **pipeline/overlap** the score computation and weighted-sum, because both read `kv_nope` sequentially — a **single Triton kernel** that reads `kv_nope` tiles **once** and simultaneously accumulates both `attn_scores` (via `q_nope_latent`) and the final value output (via the online-softmax trick, reusing score values to update a running weighted sum) would halve the HBM traffic on the dominant tensor.

## PROPOSAL
Implement a **fused MLA attention kernel** in Triton that processes the latent KV cache in a single streaming pass, using the **online-softmax (Flash Attention-style)** algorithm adapted for the absorbed MLA formulation. Each CTA handles one `(batch, head)` pair. The kernel streams tiles of `kv_nope [bs, kv_len, dkv]` from HBM, for each tile computing: (1) `nope_score += q_nope_latent[b,h,:] · kv_nope[b,t,:]` and the RoPE score from `q_rope[b,h,:] · k_rope[b,t,:]` (k_rope is cheap to recompute inline from stored `kv_lora[..., dkv:]`), (2) online max/sum update for numerical stability, (3) weighted accumulation of `kv_nope[b,t,:]` with current exp-normalized weight for the value output (since V is also absorbed: `y = attn_weights @ kv_nope @ wV^T`). This approach reads `kv_nope` exactly once, avoids materializing the full `[bs×nh, kv_len]` score matrix, and eliminates the separate softmax kernel and the `M = attn @ kv_nope` matmul entirely. The result `M [bs, nh, dkv]` is then projected through `wV` as before. This is the MLA-specific analog of Flash Attention and should cut the attention phase from 4 HBM passes to 1.

## Iteration 3

## STATE
Three experiments total: baseline at 2612 µs, then two attempts at Triton-based fused attention kernels both deeply failed (60–87× slower). Experiment #3's diagnosis is definitive: a CTA-per-(batch, head) decomposition with DKV=512 register accumulator is fatally register-bound and achieves terrible occupancy. No custom Triton attention kernel has come close to the baseline yet. The critical lesson is that the baseline's cuBLAS GEMMs (`scores_nope = q_nope_latent @ kv_nope^T` as [bs×nh, 1, dkv] × [bs, dkv, kv_len]) are already hardware-optimal for this shape because they exploit cuBLAS's batched GEMM tiling and can tile dkv=512 in SRAM efficiently using tensor cores. Custom Triton kernels with per-CTA streaming cannot compete with cuBLAS here.

## RATIONALE
The baseline is spending time across multiple serialized ops with unnecessary HBM round-trips, but the individual ops are well-chosen. The correct strategy is not to replace the cuBLAS GEMMs, but to aggressively reduce overhead **around** them: eliminate Python/PyTorch dispatch overhead, eliminate intermediate tensor allocations, and look for parallelism across independent op chains. The two score paths (nope and rope) are computed sequentially but are fully independent — they can be overlapped. More importantly, the `wDQ`, `wDKV`, `wUQ` linear projections at the beginning are also sequential but `q_lora = x @ wDQ` and `kv_lora_input = x @ wDKV` are independent and could be batched into a single GEMM. The `torch.einsum('bhd,hdk->bhk', q_nope, wK)` is a batched contraction that goes through einsum dispatch rather than a direct cuBLAS call — replacing it with explicit `torch.bmm` after reshaping can remove einsum overhead. Similarly `_rotate_half` creates two extra allocations. Using `torch.cuda.streams` to overlap independent computations is another axis. Finally, `torch.compile` with `mode='max-autotune'` should be tried — it can fuse elementwise ops, eliminate temporaries, and overlap kernels automatically without any manual kernel writing.

## PROPOSAL
Try **`torch.compile` with `mode='max-autotune'`** on the entire `custom_kernel` function, keeping the baseline's absorbed-key algorithm intact. Wrap the core compute (everything after the KV cache update) in a compiled function. `torch.compile` with max-autotune will: (1) fuse the elementwise RoPE ops and eliminate intermediate tensor allocations, (2) potentially batch the two independent score GEMMs, (3) eliminate Python overhead on repeated calls. Use `torch.compile(..., mode='max-autotune', fullgraph=True)` if possible, or fallback to `mode='reduce-overhead'`. The key is the first warm-up call will be slow but benchmarks run after warm-up, so compilation cost is amortized. Also replace the `torch.einsum` calls with explicit `torch.bmm` reshapes to give the compiler (and cuBLAS) cleaner GEMM shapes. If `torch.compile` is not viable (e.g., graph breaks due to dynamic shapes), fall back to replacing `_rotate_half` with an in-place elementwise op and batching `wDQ`/`wDKV` into a single wider GEMM to reduce kernel launch overhead.

## Iteration 4

## STATE
Four experiments total: baseline at 2612 µs holds. Three custom approaches all failed badly — SDPA with full K/V expansion (227k µs), Triton flash-attn kernel with DKV=512 register pressure (167k µs), and torch.compile (10k µs). The compile failure confirms dynamic `kv_len` causes recompilation overhead or graph breaks on every call. No approach has yet beaten the baseline. The baseline's cuBLAS GEMMs for the two score computations are the right primitive; the opportunity is reducing the 4-pass HBM traffic on `kv_nope` without moving to per-CTA streaming.

## RATIONALE
Looking more carefully at the baseline: the bottleneck is three sequential HBM reads of `kv_nope [bs=128, kv_len=6144, dkv=512]` which is 128×6144×512×2 = 800 MB: once for `scores_nope = q_nope_latent @ kv_nope^T` (reads kv_nope), once writing `attn_flat` (not kv_nope but score matrix), and once for `M = attn @ kv_nope` (reads kv_nope again). These two kv_nope reads are the dominant cost — 1.6 GB of HBM traffic just for kv_nope, on an H200 with ~3.35 TB/s bandwidth = ~480 µs theoretical minimum for kv_nope alone. The `fused_add_scale_softmax` kernel also reads two [bs×nh, kv_len] = 128×128×6144×2 ≈ 200 MB tensors. The GEMM-based approach must be kept (cuBLAS tiling is optimal) but the two separate kv_nope reads could be eliminated by **precomputing `k_rope` offline** (it doesn't depend on heads) and looking at whether the `scores_rope` GEMM can be combined with anything. The most impactful single change is to **precompute and store `k_rope` in the KV cache** — but we can't modify the cache structure. However, we can precompute `k_rope` once per call (currently done), and recognize that `scores_rope [bs, nh, kv_len]` is shared across all heads (k_rope is head-independent). This means `scores_rope` can be computed as `[bs, 1, kv_len]` and broadcast — already being done as `matmul(q_rope [bs,nh,64], k_rope^T [bs,64,kv_len])`. The nope path reads kv_nope twice; fusing those two reads into one pass saves 800 MB of HBM traffic. The correct approach: replace `(scores_nope = q_nope_latent @ kv_nope^T)` followed by `(M = attn @ kv_nope)` with a **single cuBLAS-backed computation** that shares the kv_nope read. This is exactly what the Triton kernel attempted but failed due to register pressure — the solution is to use a **split-K / output-tiling strategy** where each CTA handles a small chunk of the DKV dimension rather than all 512 dimensions, reducing register pressure dramatically.

## PROPOSAL
Revisit the fused MLA Triton kernel, but fix the register pressure problem by **tiling the DKV=512 accumulator dimension** across multiple CTAs. Instead of one CTA per (batch, head) holding 512 float32 accumulators, split DKV into small tiles (e.g., 64 elements per CTA) so each CTA holds only a 64-float32 accumulator. The grid becomes `[bs × nh × (DKV / TILE_DKV)]` CTAs. Each CTA handles its slice of the DKV dimension for both: (a) computing its contribution to `score_nope` (dot product of `q_nope_latent[b,h, dkv_tile]` with `kv_nope[b,t, dkv_tile]` — this is a partial dot, needs a final reduction across tiles), and (b) accumulating `M[b,h, dkv_tile] = sum_t(attn_t * kv_nope[b,t, dkv_tile])`. The score reduction requires an inter-CTA reduction step (write partial nope scores to a scratch buffer, reduce, then do softmax, then weighted-sum), which adds complexity. An alternative is to **split only the output (value accumulation) across CTAs but keep scores serial**: keep score computation as-is via cuBLAS (which already tiles well), and only replace the `M = attn @ kv_nope` step with a tiled Triton kernel that reads `kv_nope` once and simultaneously reads the `attn` weights — saving the second kv_nope read while avoiding the score computation entirely in Triton. This means: keep `scores_nope = q_nope_latent @ kv_nope^T` as cuBLAS, keep `fused_add_scale_softmax` as-is, then replace `M = attn @ kv_nope` with a tiled Triton GEMV kernel that tiles DKV (not T) per CTA to keep register pressure low. Each CTA handles `[TILE_DKV]` elements of the output accumulator, loads full `attn[b,h,:]` and corresponding `kv_nope[b,:,dkv_slice]`, and produces `M[b,h,dkv_slice]`. This gives low register pressure (TILE_DKV floats + BLOCK_T attn weights) and reads kv_nope once in a streaming pattern.

## Iteration 5

## STATE
Five experiments, all custom kernels worse than the 2612 µs baseline. The clear pattern: every Triton kernel attempted has been 13–90× slower than cuBLAS. Custom Triton cannot replicate tensor-core GEMM performance for these shapes. The baseline is structurally sound and uses the correct absorbed-key algorithm with cuBLAS for the heavy GEMMs. The field of custom attention kernels is exhausted for this setup without a fundamentally different approach. The 10× gap to SOL (210–280 µs) must come from somewhere else: likely the **GEMM chain itself** — there are 5 sequential F.linear/matmul calls, each with kernel launch overhead and no batching.

## RATIONALE
The baseline has these sequential operations: (1) `x @ wDQ` [128,1,7168]→[128,1,1536], (2) `x @ wDKV` [128,1,7168]→[128,1,576], (3) `q_lora @ wUQ` [128,1536]→[128,21504], (4) `q_nope_latent @ kv_nope^T` [128,128,512]×[128,512,kv_len], (5) `q_rope @ k_rope^T` [128,128,64]×[128,64,kv_len], (6) `attn @ kv_nope` [128,128,kv_len]×[128,kv_len,512], (7) `M @ wV_T` [128,128,512]×[128,512,128], (8) `y @ wO` [128,1,16384]→[128,1,7168]. Steps (1) and (2) share the same input `x` and can be **batched into a single wider GEMM** by concatenating `wDQ` and `wDKV` along the output dimension. Steps (4) and (5) compute independent scores that are summed — they could be issued to separate CUDA streams to overlap. The `torch.einsum('bhd,hdk->bhk', ...)` calls for steps (3) and (7) go through a slower path than `torch.bmm`; replacing them with explicit `bmm` after reshaping eliminates einsum dispatch. The key insight: the **entire baseline is CPU-bound on kernel dispatch** — each PyTorch op launches a separate CUDA kernel, and with 8+ launches the ~10-20 µs/kernel overhead adds up significantly (potentially 150+ µs overhead alone). Using `torch.cuda.CUDAGraph` to capture and replay the entire sequence would eliminate all Python dispatch overhead while keeping cuBLAS for the GEMMs.

## PROPOSAL
**Use CUDA Graphs** to capture the entire attention computation. The key: CUDA Graphs capture the kernel launch sequence once and replay it with near-zero CPU overhead. Implement a `CUDAGraphRunner` class that: (1) on first call, allocates static input/output buffers, runs a warmup, captures the graph; (2) on subsequent calls, copies fresh data into the static buffers and replays the graph. Handle the dynamic `kv_len` by keeping `max_seq_len` as the static dimension and masking with a padding approach, OR by maintaining a cache of graphs keyed by `kv_len`. Use `torch.cuda.graph` context manager. This is a pure Python/orchestration change that keeps every single cuBLAS kernel intact — it only eliminates the Python dispatch overhead between launches. Additionally, replace `torch.einsum` calls with `torch.bmm` + reshape to remove einsum overhead, and batch `wDQ`/`wDKV` projections into a single GEMM by stacking the weight matrices.

## Iteration 6

## STATE
Six experiments; baseline holds at 2612 µs, experiment #6 (CUDA graphs) was essentially a dead heat at 2655 µs confirming there is negligible Python dispatch overhead to save. The experiment map is now clear: custom Triton kernels are all 13–90× slower; torch.compile with dynamic kv_len triggers recompilation; CUDA graphs OOM on dynamic shapes; SDPA with full K/V expansion is catastrophically memory-hungry. The baseline's cuBLAS matmuls are the performance ceiling for individual ops. Yet we remain ~10× from SOL. Something fundamentally structural must be rethought.

## RATIONALE
The near-identity result from CUDA graphs (#6) is highly informative: it confirms the bottleneck is **not** CPU-side dispatch but pure GPU compute/memory latency. Given the H200 has ~3.35 TB/s HBM bandwidth and the roofline SOL is ~210 µs, and the baseline is 2612 µs, the baseline is running at roughly 8% of peak bandwidth. This huge gap can only be explained by one thing: **the baseline is not bandwidth-bound — it's doing more FLOPs than necessary**. The key ops are: `einsum('bhd,hdk->bhk')` for `q_nope_latent` — this is a batched matmul `[bs, nh, d_nope=128] × [nh, d_nope, dkv=512]` which has bs×nh×d_nope×dkv = 128×128×128×512 = 1.07 billion FLOPs per call; and `einsum('bhd,hdk->bhk')` for `y_head` = `M @ wV_T` — same shape/FLOPs. The `scores_nope` matmul `[bs×nh, 1, dkv] × [bs, dkv, kv_len]` = 128×128×512×6144 = 51 billion FLOPs. This computation is correctly the dominant term and cuBLAS is using tensor cores. But the issue is `wK` and `wV` are being recomputed per call from `wUKV` — the slice and view operations create non-contiguous tensors that may hurt cuBLAS performance. More critically: **can `wDQ` and `wDKV` be concatenated into one wider GEMM**, and **can `wUQ` and `wO` be similarly merged**? The two independent GEMMs `x @ wDQ` and `x @ wDKV` share the same input `x [128, 1, 7168]`; doing them as one GEMM `x @ [wDQ; wDKV]` halves the number of times `x` must be read. This saves 128×1×7168×2 bytes = 1.8 MB per call — small but combined with the `wUQ` GEMM saving, could cut early-stage latency. The **real opportunity**: the weight matrices `wK [nh=128, d_nope=128, dkv=512]` are sliced from `wUKV` which is stored contiguously — but the einsum needs it in a non-standard layout. Pre-transposing and materializing `wK_T [nh, dkv, d_nope]` and `wV [nh, dkv, dv]` as **contiguous tensors cached between calls** would eliminate transposition overhead and improve cuBLAS memory access patterns.

## PROPOSAL
**Precompute and cache all weight projections** in a module-level dict keyed by weight tensor identity, eliminating repeated slicing, viewing, and transposing of weight matrices on every forward call. Specifically: (1) Cache `wK [nh, d_nope, dkv]` = `wUKV_view[:, :d_nope, :]` as a **contiguous tensor** using `.contiguous()`; (2) Cache `wK_T [nh, dkv, d_nope]` = `wK.transpose(1,2).contiguous()` which is needed for the q_nope_latent einsum; (3) Cache `wV_T [nh, dkv, dv]` = `wUKV_view[:, d_nope:, :].transpose(1,2).contiguous()`; (4) Replace both `torch.einsum('bhd,hdk->bhk', ...)` calls with explicit `torch.bmm` after reshaping — `q_nope_latent = torch.bmm(q_nope.reshape(bs*nh, 1, d_nope), wK_T.unsqueeze(0).expand(bs, -1,-1,-1).reshape(bs*nh, dkv, d_nope)).reshape(bs, nh, dkv)` — but more efficiently, pre-cache `wK_T` reshaped as `[1, nh, dkv, d_nope]` and use `torch.matmul` with proper broadcasting; (5) Also cache `[wDQ_T; wDKV_T]` concatenated so `x @ wDQ` and `x @ wDKV` are done in one wider GEMM. These are pure layout/caching changes with no algorithmic modification, targeting the overhead of repeated non-contiguous tensor creation.

## Iteration 7

## STATE
Seven experiments, baseline 2612 µs still holds. Experiments #6 and #7 both came in at 2655 µs — essentially identical to baseline, confirming the bottleneck is pure GPU compute, not CPU dispatch, not tensor layout, not weight slicing. The three most informative results: CUDA graphs = no gain (not dispatch-bound), weight caching = no gain (not layout-bound), and all custom Triton attention kernels = 13–90× slower (not improvable with manual GEMV). We are firmly GPU-compute-bound on the cuBLAS GEMMs.

## RATIONALE
At this point the evidence points to one unexplored avenue: **`flash_attn` package**. The `flash_attn` library ships a highly optimized `flash_attn_with_kvcache` function specifically designed for the decode pattern (single query token, long KV cache). Unlike SDPA which requires materializing full per-head K/V, `flash_attn_with_kvcache` accepts the KV cache in a packed format. However, MLA's absorbed-key trick means K is not stored per-head — it's in latent space. A different `flash_attn` entry point, `flash_attn_func`, can be used if we pass the full Q `[bs, 1, nh, head_dim]` and K/V `[bs, kv_len, nh, head_dim]` — but that requires materializing per-head K/V which we know is catastrophic.

The real insight missed so far: the `scores_rope` computation uses `k_rope [bs, kv_len, d_rope=64]` which is **head-shared** — the same k_rope is used for all 128 heads. The baseline broadcasts this via `matmul(q_rope [bs, nh, 64], k_rope^T [bs, 64, kv_len])` which PyTorch handles via batch broadcasting. But `scores_nope` uses batched GEMM `[bs, nh, 1, dkv] @ [bs, 1, dkv, kv_len]` — this is actually a GEMM where each of bs×nh rows is doing the same dot product against `kv_nope[b, :, :]` which is batch-b-specific but head-shared. The **key algebraic identity** that the baseline exploits but may not exploit fully: `scores_nope[b,h,t] = q_nope_latent[b,h,:] · kv_nope[b,t,:]`. This is a batched matmul where the KV side doesn't vary across heads. Could we reshape to exploit this? `q_nope_latent [bs, nh, dkv]` × `kv_nope^T [bs, dkv, kv_len]` = `scores_nope [bs, nh, kv_len]`. This is exactly what `torch.matmul` does with broadcasting — it should already be issuing one batched GEMM per batch element with nh rows. This is correct and optimal.

The remaining untested option: **`xformers` memory-efficient attention** or **`flash_attn` with the absorbed key trick**. In DeepSeek's own implementation, they use a custom CUDA kernel called `flash_mla` which handles the latent KV directly. The key trick: instead of materializing per-head K/V, they pass `c_kv [bs, kv_len, dkv]` as the "key" with a "block_table" and a custom head_dim that equals dkv=512. This is essentially standard flash attention with `head_dim=512` and `num_kv_heads=1` (since k_rope needs separate handling). If `flash_attn` supports `head_dim=512`, this would be the right path.

## PROPOSAL
Try using **`flash_attn.flash_attn_func`** or **`flash_attn.flash_attn_with_kvcache`** with the following reformulation: treat the problem as attention with `num_heads=128`, `num_kv_heads=1` (GQA/MQA), `head_dim=dkv=512` for the nope portion, handling the rope scores separately. Specifically: (1) compute `q_nope_latent [bs, nh, dkv]` as before; (2) compute `scores_rope [bs, nh, kv_len]` separately via standard matmul; (3) call `flash_attn_func` with Q=`q_nope_latent.unsqueeze(2)` `[bs, 1, nh, dkv]`, K=`kv_nope_input.unsqueeze(2)` `[bs, kv_len, 1, dkv]` (GQA with 1 KV head), V=`kv_nope_input.unsqueeze(2)` `[bs, kv_len, 1, dkv]`, with an **attention bias** of `scores_rope * scale` added — but flash_attn doesn't support arbitrary attention bias in GQA mode. Alternative: add `scores_rope` to the scores via the `softmax_scale` trick won't work for additive bias. **Simplest valid approach**: check if `flash_attn` is available and if it supports `head_dim=512` (H200 may support this), and if so, run absorption as `Q [bs, 1, nh, dkv]`, K=`kv_nope [bs, kv_len, 1, dkv]` (GQA), V=`kv_nope [bs, kv_len, 1, dkv]` (GQA), passing `scores_rope` as `alibi_slopes` — not correct. **Most honest approach**: just try importing `flash_attn` and using `flash_attn_with_kvcache` on the nope portion alone, separately summing the rope contribution to scores via a post-hoc correction, and measure whether flash_attn's fused kernel beats two separate cuBLAS GEMMs for the `scores_nope` + `M` computation.

## Iteration 8

## STATE
First keep since iteration 1: experiment #8 achieved 2587 µs, a ~1% improvement over the 2612 µs baseline using `flash_attn_func` with GQA (1 KV head, head_dim=512) and `scores_rope` as `attn_bias`. This is a small but real gain, confirming `flash_attn` is available and functional on H200. The approach is still immature — the current implementation has correctness that likely introduced overhead (e.g., `scores_rope * scale` is applied twice since `softmax_scale=scale` is also passed to flash_attn). The gain of ~24 µs is near the noise floor. The SOL gap remains large (~2400 µs away). The `flash_attn` path is the right direction and needs to be matured.

## RATIONALE
The current flash_attn usage in #8 has a correctness issue that likely hurts performance: `attn_bias = scores_rope * scale` is computed with scale, then `softmax_scale=scale` is passed to flash_attn which scales Q·K^T by scale again. This means the nope scores get scaled correctly (`q_nope_latent · kv_nope * scale`), but the rope scores in attn_bias are already pre-scaled by `scale` and then NOT re-scaled by flash_attn (attn_bias is added after the Q·K^T scaling). This is mathematically correct by accident only if flash_attn adds attn_bias to the pre-scaled scores. More importantly, there is a critical **dead computation**: `scores_rope` is computed with `q_nope_latent @ kv_nope * scale` happening inside flash_attn, but `scores_rope = torch.matmul(q_rope, k_rope.transpose(-2, -1))` is computed outside and added as bias — this is the right approach but the `scores_rope` matmul is still a separate GPU kernel. The larger opportunity: `k_rope` computation (`k_rope_input * cos_k + _rotate_half(k_rope_input) * sin_k`) creates a `[bs=128, kv_len=6144, 64]` = 100MB tensor via elementwise ops, then `scores_rope` = `[bs,nh,kv_len]` via matmul. These are two extra kernel launches and 100MB of temp memory that could be eliminated. The current code also still has the `wDQ_wDKV` fused GEMM from #7 which gave no gain — worth removing in case it hurts. The most impactful next step: **use `flash_attn_with_kvcache`** which is specifically designed for decode (single query against long KV) and avoids materializing the full KV in a temporary tensor, or alternatively **precompute `scores_rope` more efficiently** using the fact that k_rope is head-independent (compute once, broadcast).

## PROPOSAL
Build on the #8 approach (flash_attn GQA with attn_bias) with two focused improvements: (1) **Fix the softmax_scale**: pass `softmax_scale=scale` to flash_attn for the Q·K nope product, and pass `attn_bias = scores_rope * scale` — but verify this is what flash_attn actually expects (bias added to pre-scaled scores). The correct formulation is `softmax((q_nope_latent · kv_nope + scores_rope) * scale)`, so attn_bias should be `scores_rope` (unscaled) and flash_attn should multiply the combined score by `scale` — but flash_attn's `softmax_scale` is applied only to Q·K, not to attn_bias. So the correct call is: `attn_bias = scores_rope * scale` (pre-scaled), `softmax_scale = scale` (for Q·K), which gives `softmax(q·k*scale + scores_rope*scale) = softmax((q·k + scores_rope)*scale)` — this is correct. (2) **Eliminate the `_rotate_half` allocation** for k_rope computation: instead of `torch.cat((-x[..., half:], x[..., :half]))` which allocates and copies, use `torch.stack` or index-based view that avoids the extra allocation. Or better: precompute k_rope in a single fused Triton kernel from the stored `k_rope_input` and `cos/sin` tables — this is cheap since d_rope=64 is small. (3) **Drop the dead weight-caching code from #7** and `_mla_flash_attn_kernel` since they're not being used, to reduce code complexity and potential overhead from dict lookups. Keep the flash_attn path clean and focused.

## Iteration 9

## STATE
Nine experiments total. Best is 2587 µs from #8 (flash_attn GQA with attn_bias). Experiment #9 came in at 2607 µs — slightly worse, within noise, effectively matching #8. The flash_attn approach is the right direction but has plateaued at ~1% improvement over baseline. The approach is young (only 2 attempts) and still has unexplored knobs. The critical observation: the `attn_bias = scores_rope * scale` is computed as a full `[bs, nh, 1, kv_len]` tensor (128×128×1×6144×2 = 200MB) materialized to HBM before being passed to flash_attn. This allocation itself may be a significant cost. Additionally, the `scores_rope` matmul itself is still a separate cuBLAS call.

## RATIONALE
The key insight that hasn't been exploited: **the `attn_bias` in flash_attn must be stored as `[bs, nh, 1, kv_len]` float16/bfloat16**, which is a 200MB tensor. Computing this requires: (1) compute `k_rope [bs, kv_len, 64]` via elementwise ops (100MB output), (2) compute `scores_rope [bs, nh, kv_len]` via matmul (800MB read of k_rope + q_rope, 200MB output), (3) multiply by `scale` and reshape. Steps 1–3 cost ~1.1GB of HBM traffic just to produce the attn_bias, and this is extra overhead that wasn't in the baseline. The baseline's `_fused_add_scale_softmax` is a Triton kernel that reads `scores_nope` and `scores_rope` and writes the attention weights — it's already quite efficient. **The flash_attn gain comes purely from fusing the M matmul** (avoiding writing/reading the full `[bs×nh, kv_len]` attention weights to HBM for the `M = attn @ kv_nope` GEMM). The overhead of computing `scores_rope` + `attn_bias` nearly cancels this gain.

The most impactful remaining optimization: **eliminate the standalone `scores_rope` computation entirely by folding it into flash_attn's Q·K product**. This can be done if we pass `q_rope` and `k_rope` as part of a combined Q/K where `Q_combined = [q_nope_latent || q_rope]` and `K_combined = [kv_nope || k_rope]` with `head_dim = dkv + d_rope = 512 + 64 = 576`. Then flash_attn computes `Q_combined · K_combined^T * scale = (q_nope_latent · kv_nope + q_rope · k_rope) * scale` in one fused pass — exactly what we need. This eliminates the separate `scores_rope` matmul AND the `attn_bias` tensor allocation entirely. K is `[bs, kv_len, 1, 576]` (GQA, 1 KV head) and V is `[bs, kv_len, 1, dkv=512]` (different head_dim for K vs V is not supported in standard flash_attn). Alternative: use `head_dim=576` for both K and V by zero-padding V with 64 zeros, or use `head_dim=dkv=512` for V by not using GQA and instead handling the rope portion separately.

## PROPOSAL
**Combine the nope and rope components into a single flash_attn call with `head_dim = dkv + d_rope = 576`**. Concatenate Q as `[q_nope_latent || q_rope] [bs, 1, nh, 576]`, K as `[kv_nope || k_rope] [bs, kv_len, 1, 576]` (GQA with 1 KV head — note k_rope is already head-shared), and V as `[kv_nope || zeros_64] [bs, kv_len, 1, 576]` (pad with zeros so head_dim matches, the extra 64 output dims are discarded). This single flash_attn call computes `softmax((q_nope·kv_nope + q_rope·k_rope) * scale) @ V` = `M` in the first 512 output dims. This eliminates: the separate `scores_rope` matmul, the `_rotate_half` allocation, and the `attn_bias` tensor — all three intermediate results. If flash_attn doesn't support different K/V head_dim (it requires them equal), then pad V to 576 and take only the first 512 output dims of `M`. This reduces HBM traffic significantly: k_rope is included in the K tensor (eliminating the separate k_rope computation) and there's no attn_bias materialization.
